Architecture

One primitive, one bus, one security model

AbyssBSD adopts BeOS's defining idea: a single message primitive carries everything — UI events, inter-thread traffic, IPC, the display protocol, and capabilities. It is the bus.

The unified message primitive

A looper is a thread with a message queue; handlers are the objects it dispatches messages to — the BeOS BLooper/BHandler model. The primitive is used three ways, and only the third involves a wire format at all:

  • In-process — a message is a value moved by ownership through a looper's queue. No serialization.
  • Inter-thread — the same: a value moved between looper threads.
  • Inter-process — the message is serialized into an envelope; the wire format exists only here.

The envelope

The cross-process representation is a universal envelope: a header (interface id, method/type id, flags, payload length, handle count), the serialized payload, and a handle array. Handles are file descriptors (passed via SCM_RIGHTS) or bus tokens, each carrying its Capsicum rights mask. Large data never travels inline — it is shared as a memory handle; dmabuf buffer sharing is exactly this case.

┌──────────────────────────────────────────────┐ │ header: interface id · method/type id │ │ flags · payload len · handle count │ ├──────────────────────────────────────────────┤ │ payload: the serialized message body │ ├──────────────────────────────────────────────┤ │ handles: [ {kind, value, cap_rights} ] … │ └──────────────────────────────────────────────┘

Self-describing payloads, typed views

The payload is BMessage-like: a self-describing structure of named, typed fields. That is what makes applications scriptable — a script can build and inspect messages with no compile-time knowledge of them. AbyssBSD's own code is not written against an untyped dict, though: Rust derive and procedural macros generate typed views over the dict, so OS code gets compile-time field checking while scripts work the dict directly.

Addressing is authority

There is no namespace of destinations. You send to a capability you hold — cap.send(msg). Capabilities are statically typed by interface: a Cap<Display> only accepts display-protocol messages. Replies ride the same mechanism — a request carries a reply-to capability.

The looper as an async executor

A looper is a thread, a message queue, and a cooperative async executor. A handler may be async and may .await. Awaiting a request/reply call suspends the handler, never the looper's thread — so "the UI thread never blocks" becomes mechanically true. While one invocation of a handler is suspended, the looper runs other handlers, never re-entering the same one: BeOS's BHandler invariant kept, with async added beneath it.

The looper & service framework is the chief structural piece the project builds for itself — a first-party crate providing loopers, handlers, typed point-to-point rings, and a transport-agnostic API. The broker realizes the whole system on this framework: it pre-creates every ring and spawns each component looper, the eager, pre-wired, statically-auditable authority graph.

Display & input

The compositor

Written from scratch, the compositor plays the role BeOS's app_server played: the single display server, speaking the native message primitive. It talks to DRM/KMS directly via the DRM ioctl uAPI — no libdrm. A render-backend seam exposes three backends:

BackendRole
CPU / dumb-bufferNeeds zero GPU stack — the zero-Mesa floor; the system boots and renders in a VM with no GPU stack at all.
GLES (OpenGL ES 3.x via EGL)The accelerated path and the M2 target; covers all hardware Mesa accelerates.
Vulkan post-v1A second accelerated backend behind the same seam.

The display protocol — native, not Wayland

Because there is one unified message primitive, the display protocol is not a separate Wayland wire format. The compositor speaks the native primitive directly. Apps render their own buffers and share them as dmabuf handles carried in messages. Key properties:

  • Surfaces & roles — a client creates a surface and gives it a role (top-level, popup, fullscreen).
  • API-agnostic buffers — GLES, Vulkan, and CPU-rendered buffers are identical to the protocol; pixel data is never inline.
  • Explicit synchronization — every buffer carries acquire and release timeline-semaphore points; implicit fencing is not used.
  • Server-side decorations — the compositor draws every title bar and frame; title-bar drags and window buttons are handled compositor-side with no client round-trip, so window manipulation is frame-perfect.
  • Authorized clipboard & drag-and-drop — the user's gesture is the capability; no client snoops the clipboard ambiently, unlike X11.
  • Screen capture — reading pixels a client did not draw is a capability, not an ambient power: scoped to a surface, an output, or the whole desktop, revocable, and authorized by user gesture. A live capture is always shown in the status-indicator area — capture is never silent.

Managed direct scanout. When a full-screen surface covers an output exactly, the compositor page-flips the client's buffer directly to KMS — no composition pass, no copy — for near-exclusive-fullscreen performance. It keeps KMS ownership: the instant an overlay is needed (notification, alt-tab, lock screen) it resumes compositing seamlessly. A game does nothing special — it uses an ordinary Vulkan swapchain.

The input service

The input service turns hardware input devices into one normalized event stream using libinput. Its interface is internal and one-directional — input service → compositor; clients receive input via the display protocol. Each key event carries both the raw keycode (the physical key, which games need) and the cooked interpretation (keysym, modifiers, text). Keyboard interpretation lives here, in the one component whose job is interpreting input hardware.

Window management — tiling and floating

Window management is the compositor's job — it owns placement, focus, and stacking. How it places windows is a selectable layout policy, and AbyssBSD ships two, first-class, from early on:

  • Tiling — Sway/i3-grade: windows fill the output without overlap, split into containers and workspaces, navigated and resized by keyboard through configurable key-chords. The power-programmer-first experience — and the first to exist, since it needs only the compositor and a keyboard, no toolkit and no pointer-driven furniture.
  • Floating — conventional overlapping windows, pointer-placed. The shipped default: the GNOME 2 desktop most users get. Tiling is the first-class alternative they may select.

Both are policies over one shared window-management core — the output coordinate space, workspaces, surface roles, focus. The GNOME 2 desktop is not a second mechanism; it is the floating policy plus the shell's furniture, so the tiling WM is the first consumer of the permanent core the desktop later reuses. Key bindings are configuration, not compiled in — chord sequences and i3-style "modes" included — and the tiling bar is the desktop shell in a minimal configuration, the GNOME 2 panels the same component, fuller.

Restraint. The tiling experience is held to the same lens as everything else — Sway/i3 in spirit: fast, keyboard-first, a small and well-specified feature set, and deliberately not Hyprland-style maximalism. "Power-user-first" describes the input model — it is not an invitation to a thousand knobs.

The toolkit

Retained-mode, organized as Kits — the BeOS structure:

KitResponsibility
Interface KitWidgets, layout, theming, the view hierarchy.
Application KitApp lifecycle, the looper/handler model, messaging.
Storage KitFiles, XDG-resolved locations; typed attributes and live queries post-v1
Media KitAudio stream handling and device re-routing later

Each window is its own looper on its own thread. The toolkit themes the widgets to the GNOME 2 appearance; window decorations are drawn by the compositor and the panel by the shell — all three draw from one shared theme so the desktop is visually coherent.

An arena of views, addressed by handle

A window's view hierarchy is not a tree of pointers or reference-counted nodes. One arena per window owns every view; every cross-reference is a ViewId — a generational handle (index + generation, the standard slotmap pattern). Resolving one generation-checks: a ViewId outliving its view resolves to none — safe and observable, never a dangling pointer, never a leak. There are no ownership cycles and no Rc/Arc. Retained-mode means only the dirty ViewId set is re-laid-out and repainted.

The view tree stores no callbacks: a widget interaction is routed to a ViewId and emitted as a message to the app's looper, not a stored closure.

The object-capability security model

AbyssBSD uses an object-capability (ocap) model. It is not a subsystem bolted on beside the message bus — it is the bus.

  • No ambient authority. A process is born holding nothing. It cannot name or reach any resource it was not given.
  • Authority is an unforgeable handle. To hold a reference to an object is the permission to use it.
  • Authority travels only in messages. The unified primitive already carries handles, so the security model and the IPC model are the same mechanism.
  • Attenuate and delegate, never amplify. A holder may pass on a handle, or a weaker derivation, but never manufacture more authority than it was given.

This is the structural reason AbyssBSD rejects D-Bus: D-Bus is a names-and-methods bus with no notion of authority, so security has to be bolted alongside it (polkit, portals, Flatpak). A capability bus needs none of them.

Enforcement — native Capsicum + jails

ConcernMechanism
No opening resources by nameCapsicum capability mode (cap_enter) — irreversible, kernel-enforced
Per-handle rightsCapsicum cap_rights_limit — fine-grained, kernel-enforced fd rights
Isolation containerjails
Process handleprocess descriptors (pdfork)
Handle transportSCM_RIGHTS for fd-backed capabilities; bus tokens for service objects

Because FreeBSD's Capsicum enforces per-fd rights in the kernel, both backings of a capability are genuinely enforced — there is no "advisory mask on a raw fd" gap. This is the concrete payoff of choosing FreeBSD over Linux.

Why FreeBSD — and, today, only FreeBSD. This is not a portable design wearing a FreeBSD coat. Kernel-enforced per-handle rights — Capsicum's cap_rights_limit — ship on exactly one general-purpose OS. A Linux port is buildable: namespaces and cgroups out-isolate jails, pidfd stands in for pdfork, memfd and SCM_RIGHTS are native. But Linux has no per-fd rights mask; seccomp can approximate capability mode and not per-handle rights, so a capability there is a sandbox with an advisory mask — AbyssBSD-shaped, not AbyssBSD. Fuchsia's Zircon kernel fits the model better than FreeBSD does — every handle carries rights bits by construction — but it offers no whole, mature base to keep and track upstream. FreeBSD is the single place a real kernel capability primitive and a complete desktop base meet. The choice was the only one on the table.

The broker

A broker is the sole minter of authority — the only component that opens devices and creates jails. It hands each child its initial capability set at spawn time; a child cannot open those resources itself, because capability mode forbids it. The broker is started early by rc and is modeled on FreeBSD's libcasper. Any capability holder can sub-broker for its own children, carving sandboxes out of its own authority.

Honest scope. The FreeBSD base sits in the trusted computing base regardless. This is not seL4-grade verified isolation, and AbyssBSD does not claim it. What the model buys is a desktop layer in which no application holds ambient authority — a strict, real improvement over every conventional desktop, now with kernel-enforced capabilities rather than a best-effort approximation.
