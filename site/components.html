Components

The component map

AbyssBSD decomposes into the components below — and the discipline is that this list is closed and curated, not open-ended. Each is one process, jailed, reached only over a bus connection carrying a capability.

No component shares memory or authority with another except by explicit handle. Granularity is by responsibility: a component is cut at one coherent responsibility. Replacing a component means supplying a process that exports the same interface. The bus itself is not a component — it is a protocol and a library; there is no central bus daemon.

Foundation

Broker

The root of authority and the session root: spawns every component, creates its jail, mints and delegates capabilities, wires bus connections, owns the session. Started by rc.

Display & input

Compositor

Owns the display (GPU/KMS), composites client surfaces, manages windows (placement, focus, stacking, decoration), and routes input to the focused surface. Exports the display protocol.

Input service

Turns hardware input devices into one normalized event stream via libinput. Exports the input-event interface, consumed by the compositor. Holds seatd-mediated device capabilities.

Login & session

Authenticator

The single trusted front-end to system authentication (FreeBSD PAM): owns the credential conversation and user-database access. Privileged, deliberately tiny, audited.

Greeter

The pre-session login UI — user picker, auth conversation, power menu. An unprivileged display client running as a dedicated _greeter principal. It renders; it does not verify.

System services — one responsibility each

ComponentResponsibility & exports
Settings service The typed settings store — get / set / subscribe.
Notification service Accepts, queues, and dispatches notifications — post / update / withdraw; the shell subscribes and renders.
Device monitor Hardware presence (hotplug, fed by devd) and removable-volume mounting.
Power & lifecycle service Suspend/resume, battery, idle detection, lock policy, shutdown coordination.
Networking Desktop network management — a control-plane orchestrator over the FreeBSD base, never in the packet path.
Audio Desktop audio control — per-app and master volume, device selection; control-plane only, the kernel mixes.
Index/query service post-v1 Indexes files and answers live queries — the BFS-like typed-attribute data model.

Shell & apps

Session lock

Draws the unlock surface and authenticates the user — a deliberately tiny, auditable component, the trusted unlock path.

Desktop shell

The in-session furniture: panels, application menu, window list, desktop surface, status-indicator area. A consumer — nothing in the system depends on it.

Apps

Terminal, file manager, settings UI, text editor. The leaves: they consume interfaces and expose only the scripting interface.

System layer vs. session layer. The map divides by lifecycle. The system layer — broker, compositor, input service, authenticator, greeter — is spawned once at boot and persists across logins. The session layer — the desktop shell, the session lock, the per-user services, and the user's apps — is spawned per login as the authenticated user and torn down at logout. The toolkit is not a component — it is a library linked into every UI process.

The desktop shell & the window list

The shell presents GNOME-2-style panels (a top and a bottom panel by default), the application menu (categorized, with a search box), the window list, the desktop surface, and a curated status-indicator area. Menus are per-window — not a global menu bar.

The window list is the anti-taskbar, held to six disciplines — the Windows taskbar being the cautionary tale of what a window list becomes without them:

  1. One button per window — never grouped; reaching a window is one click.
  2. The button shows the window's title, and nothing else.
  3. Launching is not here — the application menu launches. No pinned apps.
  4. No tray. The status-indicator area is a curated, fixed set — clock, battery, network, volume — apps cannot dump icons into it.
  5. No search box, no widgets, no "recommended," no ads — not ever.
  6. The model is stable; it does not churn across versions.

The app model

An AbyssBSD app is a bundle — a directory presented as one object, in the macOS .app tradition. The bundle holds an app manifest, the executable, resources, and the libraries the app needs. The manifest declares identity, the capabilities the app requests, and its library dependencies by content hash.

  • Libraries are content-addressed. Each library is stored once, system-wide, keyed by the hash of its contents. The store cannot hold a duplicate — "no duplication" by construction, and no DLL hell.
  • Dedup serves the memory budget. Because every app references the identical library file, the kernel maps its text once and shares it across all of them.
  • Self-contained for transport. A shipped bundle carries its libraries; on install they merge into the store, deduplicated on the way in.
  • Capabilities are requested, not assumed. A curated app's manifest is curation-vetted; a third-party app's requests beyond a safe default are surfaced for the user to approve.

Packaging & distribution

AbyssBSD rides FreeBSD's ports + pkg — they are how the curators build the AbyssBSD layer. pkg is a build-time tool, not the on-machine update path. The product ships as a curated, installable FreeBSD-based image with the desktop preinstalled.

Install and system update are one lifecycle, both running on ZFS boot environments. An update creates a fresh boot environment, populates it, marks it active, and reboots into it — atomic; the running system is never half-updated. A freshly-activated environment is on probation: if it does not reach a healthy desktop, the system auto-reverts. The installer is a graphical application running in a live AbyssBSD environment — there is no text-mode curses installer.
