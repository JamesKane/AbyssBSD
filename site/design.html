Design

The vision & the principles

AbyssBSD is an opinionated desktop OS on the FreeBSD base: a BeOS-influenced architecture and a GNOME-2-style graphical desktop layered on top of FreeBSD's kernel, libc, drivers, toolchain, and ports.

1 · The vision

FreeBSD is kept whole — kernel, libc, device drivers, the LLVM/Clang toolchain, base utilities, rc, ports/pkg. AbyssBSD adds, on top, a coherent graphical desktop with a genuinely new architecture:

  • One unified, typed message primitive that carries UI events, inter-thread traffic, and IPC alike — the BeOS idea.
  • An object-capability security model woven through that bus, backed by FreeBSD's native Capsicum and jails.
  • A from-scratch, Wayland-free compositor.
  • A Kit-structured, retained-mode toolkit.
  • A desktop that looks like GNOME 2 and feels like BeOS — snappy, message-driven, never stalling.
  • A coherent architecture in which every component does one thing and is replaceable behind an enforced interface — coherent like macOS, but swappable.

The experience is BeOS-like — the feel, not the chrome. The visual surface is deliberately conventional, so the novelty is in the architecture, not the look.

2 · AbyssBSD vs. the FreeBSD base

AbyssBSD is a layer, not a from-scratch OS. The boundary is fixed and deliberate:

FreeBSD providesAbyssBSD provides
Kernel (Capsicum, jails, process descriptors, drm-kmod for KMS, the vt console); base libc; device drivers; the Clang/LLVM toolchain; base CLI utilities; init/rc; the shell; the ports tree and pkg. The message bus; the capability broker; the compositor / display server; the Kit toolkit; the desktop shell; the core apps; and the curation that makes it one coherent product.

AbyssBSD does not fork the FreeBSD base — it tracks it upstream. That is what makes the project an order of magnitude smaller than a new-userland OS. The capability broker runs as an rc service and is permanently desktop-scoped; rc remains the system init and supervises the broker itself.

3 · Design principles

3.1 · The implementation language — Rust

The AbyssBSD layer is written in Rust. The architecture needs a shipping, mature systems language that meets a set of hard requirements with no compiler-maturity risk — and Rust is the one that delivers all of them today: memory safety without a GC, move semantics, data-race-free concurrency checked by the compiler, C FFI, generics, compile-time codegen, and native async/.await.

Two things the architecture leans on are not language surface in Rust, so they become first-party AbyssBSD code: the looper/service framework (Rust has async but no actor model) and capability rights as typestate. Neither is a blocker; both are ordinary Rust.

3.2 · Zero vendored dependencies

The AbyssBSD-layer code leans on the Rust standard library and a curated set of first-party crates, plus an explicit, version-controlled allowlist for any external crate or build-time tool. No dependency is taken for "a few methods." FreeBSD ports the layer depends on are kept to a small, recorded set — the font stack, libinput, seatd, Mesa. Adding to either list is a deliberate decision.

3.3 · Opinionated

Strong, curated defaults and a coherent vision over endless configurability. The opinion is: GNOME 2 surface, BeOS architecture, capability security, no legacy desktop cruft.

3.4 · One thing well, replaceable at the seam

Every component does one thing, and components interact only through typed messages over the capability bus — never shared internals, never ambient authority. A component is defined by the message interface it exports, and anything exporting that interface is a valid replacement.

This is the deliberate answer to the freedesktop desktop — which is not short of small components, but short of a coherent design and of enforced boundaries. AbyssBSD takes both: the coherence of one opinionated design and genuine replaceability, because the bus and object-capabilities enforce the seam structurally.

  • No duplication. No two components share a responsibility.
  • The interface is the artifact. A component's exported message interface is a first-class specified thing.
  • Replaceable, not endlessly configurable. AbyssBSD ships one curated whole; clean seams keep a part swappable.

3.5 · The review lens

Every component is held against one yardstick — the shared sensibility of Carmack, Ousterhout, Muratori, and Blow:

  • Hold the whole thing in your head. A component too large to understand is, by that fact, a defect.
  • Earn every abstraction. A deep module is good; a shallow one, and speculative generality, are not.
  • Refuse incremental complexity. It arrives one locally-justified addition at a time — that is how systemd happened.
  • Dependencies are liabilities, each one weighed.
  • No hidden control flow, no opaque state, no magic.
  • Measure; do not guess.

3.6 · Performance & memory budgets

A 12-core, 5 GHz, RTX-5060-class machine should never feel slower than a 1995 desktop did. That it does is accreted latency. The architecture removes the causes; the budgets hold it to that.

Latency

The software AbyssBSD controls adds at most one refresh interval. The compositor finishes every frame within the refresh interval — zero dropped frames under desktop load.

Memory

The idle desktop is budgeted at the display's triple-buffer cost plus a bounded constant — on the order of 256 MB at 4K. A current GNOME/KDE desktop idles at 1–2 GB.

Idle CPU — zero

Components are event-driven; a looper wakes only on a message. An idle desktop does no work. No polling loops, anywhere.

The budgets are walls, not targets. Memory is enforced by the broker via jail resource limits. Input-to-photon is measured continuously and gates CI on the p99. Exceeding either budget is a build or runtime failure — a legitimate increase is a deliberate, reviewed manifest change. Soft tracking is how the industry drifted into the mess this section exists to prevent.

Non-binding goal — 32-bit degradability. AbyssBSD's own code is kept 32-bit-clean as a forcing function: code that genuinely fits a 4 GB address space has proven it carries no bloat.
