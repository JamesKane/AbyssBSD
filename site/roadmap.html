Roadmap

Five milestones

There is no self-host milestone — FreeBSD already provides a self-hosting system. The message primitive, the bus, and the capability broker are foundational, built across M1, because the compositor depends on them.

  • M1 Minimal framebuffer UI

    On a stock FreeBSD system, an AbyssBSD rc service starts broker → compositor (CPU backend) plus one terminal window. The message primitive, bus, and broker exist. This is the recovery floor forever after — the terminal emulator is a real VT and is load-bearing from M1.

  • M2 GPU path

    A GLES 3.x backend, Mesa, and dmabuf buffer sharing; the compositor goes accelerated.

  • M3 Toolkit + desktop

    Retained-mode Kits and the GNOME-2 shell — panel, application menu, window list.

  • M4 Core apps

    File manager, settings, text editor. The terminal already exists from M1.

  • M5 Distribution & hardening

    A curated installable image and installer; performance and daily-driver work; broadening curated hardware support.

Capsicum and jail confinement is applied per service as each is brought up through M1–M5.

The desktop experience

  • No text mode. Boot lands directly in the AbyssBSD graphical session. FreeBSD's vt console still exists beneath, as a safety valve only.
  • First boot = minimal framebuffer UI. rc → broker → compositor (CPU backend) → one terminal window.
  • The minimal framebuffer UI is the permanent recovery floor. It is simultaneously the bringup environment and the rescue environment, and uses the CPU render backend so it always comes up.

Reference hardware

AbyssBSD adds no hardware enablement of its own — supported hardware is exactly what FreeBSD's drivers and drm-kmod provide. The project is VM-first early on; the bare-metal development and test reference is a desktop with an AMD RX 6750 XT (RDNA2) — first-class amdgpu support under Mesa, well covered by recent drm-kmod.

Open threads

Vulkan backend post-v1

A second accelerated render backend behind the existing render-backend seam, added once the GLES path is solid. Note that client-side Vulkan — for games using direct scanout — ships in v1 regardless; only the compositor's own Vulkan render backend is deferred.

Typed attributes & live queries post-v1

The BeOS data-model legacy: a file is not only its bytes but a set of typed attributes, and the filesystem can be queried like a database with results that stay live. Delivered by the index/query service over ordinary FreeBSD files — the Spotlight architecture, not the BFS or WinFS one.
