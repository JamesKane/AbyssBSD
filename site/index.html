AbyssBSD

A desktop operating system that fits in your head.

An opinionated desktop OS on the FreeBSD base — the feel of BeOS, the look of GNOME 2, capability security woven through a single message bus, and not one line of garbage-collected code in the resident set.

The problem

The 40-million-line problem

A modern desktop is an act of forgetting. Boot one and you are running tens of millions of lines of code — kernel, display server, message bus, policy daemons, portal services, toolkit upon toolkit — that no single person has read, that nobody fully understands, and that grows by accretion every release.

The hardware underneath got a thousand times faster since 1995. The desktop did not get faster. It got heavier. Every gain was spent on layers. This is not a law of nature — it is a habit. AbyssBSD is the refusal of that habit.

What it is

A new desktop on a whole FreeBSD base

AbyssBSD keeps the FreeBSD base whole — kernel, libc, drivers, the LLVM/Clang toolchain, and ports — tracked upstream and never forked. FreeBSD does the unglamorous 90%, and does it well. On top of that base, AbyssBSD adds one coherent thing: a desktop with a genuinely new architecture.

It ships as a curated, installable FreeBSD-based image — the same form as helloSystem or GhostBSD — but with a new architecture above the base rather than a conventional desktop environment.

The ideas

Six commitments

One message primitive

A single typed message carries UI events, inter-thread traffic, and IPC alike. It is the bus. No D-Bus, no second mechanism.

Capabilities, not ambient authority

A process is born holding nothing. Authority travels only as unforgeable handles in messages, backed by FreeBSD's native Capsicum and jails.

One thing well, swappable

Every component does one job behind an enforced message interface. Coherent like macOS — but every part replaceable at the seam.

From-scratch compositor & toolkit

Wayland-free, retained-mode, server-side decorations, and direct scanout so full-screen games pay nothing.

Zero vendored dependencies

The AbyssBSD layer leans on the Rust standard library and a tiny, version-controlled allowlist. Every dependency is a decision.

Budgets are walls

The idle desktop is budgeted at ~256 MB at 4K. Exceeding a budget is a build failure — not a regret logged for later.

Written in Rust

No garbage collector. Anywhere.

The whole AbyssBSD layer — bus, broker, compositor, toolkit, shell, and apps — is written in Rust: memory safety without a garbage collector, compiler-checked concurrency, and a mature toolchain with no runtime to wait on. No GC pauses anywhere in the resident set.

The FreeBSD base below stays C, well-maintained upstream. Rust is chosen because it ships, today, every hard requirement the architecture needs — with no compiler-maturity risk to carry.

The wager

It should never feel slower than 1995

A 12-core, 5 GHz machine should never feel slower than a 1995 desktop did. That it routinely does is accreted latency — and accreted latency can be designed out. AbyssBSD is small enough to hold in one head, fast because it was measured, and secure because authority is explicit.

A return to simplicity. By construction.
