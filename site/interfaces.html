Interfaces

The message-interface contracts

The design gives each component interface its shape. The interface documents give the concrete typed message schema: the actual messages, their fields and types, their request/reply pairings. One document per interface.

Message kinds

Every message is one of three kinds:

  • request: carries a reply-to capability; the recipient sends back exactly one message: the typed success reply, or an Error.
  • command: fire-and-forget; no reply, no reply-to capability.
  • event: unsolicited; sent by a service to a retained sink capability the client gave it earlier.

A subscription is a request whose reply-to capability the service retains (rather than spending it on one reply) and uses as the event sink for an ongoing stream.

Conventions

Request / reply

A request carries a fresh reply-to capability. The capability is the correlation: there are no request-ids, and a reply can reach no one but the one waiter.

Errors are values

A request's reply is a success value or an Error, never an exception. Error is { code, detail }: code a per-interface enum, detail a human-readable string.

Lifetime

A subscription ends when the capability is dropped or its holder disconnects. No explicit teardown is needed for correctness.

Envelope mapping

Each message is one envelope: the interface id and method id in the header; the fields serialized as the payload; every capability, including the reply-to capability, in the handle array.

Notation

A message is written as a name and kind, its typed fields, and (for requests) its reply type:

Name  : kind
  field : Type
  field : Type
  → ReplyType | Error        (requests only)

Ack is the empty success reply. Types are the typed-value types (bool, i64, f64, string, enum, list<T>, dict) plus interface-defined types.

The interfaces

One document per component interface. Each has, in order: the interface identity and consumers, data types, messages grouped by direction, capabilities, errors, and accepted/rejected example exchanges.

settings store

The typed configuration store: get / set / subscribe. Schemas are declarative and shipped; access is capability-scoped to a subtree and to read or read-write. The first interface document, and the template.

input stream

The normalized input event stream, from input service → compositor. Internal and one-directional; clients receive input via the display protocol instead.

notification service

Posting notifications and rendering them: post / update / withdraw for posters; subscribe and history for the shell. Action buttons ride reply-to capabilities.

power service

Power & lifecycle: events, control, and inhibitors. An inhibitor is a capability handle: the inhibit lasts exactly as long as the handle is held.

device-monitor service

Device presence and removable-volume mounting. Exports device arrival/change/removal events and volume mount/unmount. Fed by FreeBSD's devd.

networking service

Desktop network management: list, connect / disconnect, status, and a connection-event subscribe. Control-plane only.

audio service

Desktop audio control: per-app and master volume, device selection, audio events. Control-plane only; the kernel mixes. Capture is a separately gated capability.

display protocol

The compositor's display protocol: surfaces, roles, buffers, commit and frame callbacks, explicit synchronization, server-side decorations, and managed direct scanout.

broker spawn & grant

Spawn, supervision, and the capability bundle. Less a request/reply API than a spawn-and-grant surface, configured by a declarative manifest per component.

scripting cross-cutting

The introspect / get / set / invoke suite every handler answers. Scripting authority is capability authority: an inspect-only cap and a full-invoke cap are genuinely different grants.

session-lock report

The unlock report: on successful authentication the session lock reports to the power & lifecycle service, which owns lock state.

shell consumer

The desktop shell, a consumer that exports only the scripting interface.

toolkit library

The toolkit: a library, not a process; the scriptable window/view model addressed by ViewId.

Why this matters. The interface is the artifact: a component is defined by the message interface it exports, and anything exporting that interface is a valid replacement. The schema is the replaceability contract.
